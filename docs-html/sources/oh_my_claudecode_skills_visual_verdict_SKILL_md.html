b002

<Purpose> Use this skill to compare generated UI screenshots against one or more reference images and return a strict JSON verdict that can drive the next edit iteration. </Purpose>

b003

<Use_When>

  • The task includes visual fidelity requirements (layout, spacing, typography, component styling)
  • You have a generated screenshot and at least one reference image
  • You need deterministic pass/fail guidance before continuing edits

</Use_When>

b004

<Inputs>

  • reference_images[] (one or more image paths)
  • generated_screenshot (current output image)
  • Optional: category_hint (e.g., hackernews, sns-feed, dashboard)

</Inputs>

b005

<Output_Contract> Return JSON only with this exact shape:

b006
{
  "score": 0,
  "verdict": "revise",
  "category_match": false,
  "differences": ["..."],
  "suggestions": ["..."],
  "reasoning": "short explanation"
}
b007

Rules:

  • score: integer 0-100
  • verdict: short status (pass, revise, or fail)
  • category_match: true when the generated screenshot matches the intended UI category/style
  • differences[]: concrete visual mismatches (layout, spacing, typography, colors, hierarchy)
  • suggestions[]: actionable next edits tied to the differences
  • reasoning: 1-2 sentence summary
b008

<Threshold_And_Loop>

  • Target pass threshold is 90+.
  • If score < 90, continue editing and rerun /oh-my-claudecode:visual-verdict before any further visual review pass.
  • Do not treat the visual task as complete until the next screenshot clears the threshold.

</Threshold_And_Loop>

b009

<Debug_Visualization> When mismatch diagnosis is hard:

  1. Keep $visual-verdict as the authoritative decision.
  2. Use pixel-level diff tooling (pixel diff / pixelmatch overlay) as a secondary debug aid to localize hotspots.
  3. Convert pixel diff hotspots into concrete differences[] and suggestions[] updates.

</Debug_Visualization>

b010

<Example>

{
  "score": 87,
  "verdict": "revise",
  "category_match": true,
  "differences": [
    "Top nav spacing is tighter than reference",
    "Primary button uses smaller font weight"
  ],
  "suggestions": [
    "Increase nav item horizontal padding by 4px",
    "Set primary button font-weight to 600"
  ],
  "reasoning": "Core layout matches, but style details still diverge."
}

</Example>

<Purpose> 使用此技能將生成的 UI 截圖與一張或多張參考圖片進行比較，並回傳嚴格的 JSON 判定，以驅動下一輪編輯。 </Purpose>

b003

<Use_When>

  • 任務包含視覺保真需求（版面、間距、排版、元件樣式）
  • 你有一張生成的截圖和至少一張參考圖片
  • 在繼續編輯前，你需要可決定性的通過／失敗指引

</Use_When>

b004

<Inputs>

  • reference_images[]（一個或多個圖片路徑）
  • generated_screenshot（目前輸出圖片）
  • Optional: category_hint（例如 hackernews、sns-feed、dashboard）

</Inputs>

b005

<Output_Contract> 僅回傳 JSON，且必須使用這個精確結構：

b006
{
  "score": 0,
  "verdict": "revise",
  "category_match": false,
  "differences": ["..."],
  "suggestions": ["..."],
  "reasoning": "簡短說明"
}
b007

規則：

  • score：0-100 的整數
  • verdict：簡短狀態（pass、revise 或 fail）
  • category_match：當生成的截圖符合預期的 UI 類別／風格時為 true
  • differences[]：具體的視覺不一致之處（版面、間距、排版、色彩、層級）
  • suggestions[]：針對差異提出可執行的下一步編輯
  • reasoning：1-2 句摘要
b008

<Threshold_And_Loop>

  • 目標通過門檻是 90+。
  • 如果 score < 90，請繼續編輯，並在進行任何進一步的視覺審查前重新執行 /oh-my-claudecode:visual-verdict。
  • 在下一張截圖達到門檻之前，不要將此視覺任務視為完成。

</Threshold_And_Loop>

b009

<Debug_Visualization> 當難以診斷不一致時：

  1. 讓 $visual-verdict 作為權威判定。
  2. 使用像素層級 diff 工具（pixel diff / pixelmatch overlay）作為次要除錯輔助來定位熱點。
  3. 將像素 diff 的熱點轉換為具體的 differences[] 與 suggestions[] 更新。

</Debug_Visualization>
