b002

Team Skill

b003

Spawn N coordinated agents working on a shared task list using Claude Code's native team tools. Replaces the legacy /swarm skill (SQLite-based) with built-in team management, inter-agent messaging, and task dependencies -- no external dependencies required.

b004

The swarm compatibility alias was removed in #1131.

b005

Usage

b006
/oh-my-claudecode:team N:agent-type "task description"
/oh-my-claudecode:team "task description"
/oh-my-claudecode:team ralph "task description"
b007

Parameters

b008
  • N - Number of teammate agents (1-20). Optional; defaults to auto-sizing based on task decomposition.
  • agent-type - OMC agent to spawn for the team-exec stage (e.g., executor, debugger, designer, codex, gemini). Optional; defaults to stage-aware routing. Use codex to spawn Codex CLI workers or gemini for Gemini CLI workers (requires respective CLIs installed). See Stage Agent Routing below.
  • task - High-level task to decompose and distribute among teammates
  • ralph - Optional modifier. When present, wraps the team pipeline in Ralph's persistence loop (retry on failure, architect verification before completion). See Team + Ralph Composition below.
b009

Examples

b010
/team 5:executor "fix all TypeScript errors across the project"
/team 3:debugger "fix build errors in src/"
/team 4:designer "implement responsive layouts for all page components"
/team "refactor the auth module with security review"
/team ralph "build a complete REST API for user management"
# With Codex CLI workers (requires: npm install -g @openai/codex)
/team 2:codex "review architecture and suggest improvements"
# With Gemini CLI workers (requires: npm install -g @google/gemini-cli)
/team 2:gemini "redesign the UI components"
# Mixed: Codex for backend analysis, Gemini for frontend (use /ccg instead for this)

Architecture

b012
User: "/team 3:executor fix all TypeScript errors"
              |
              v
      [TEAM ORCHESTRATOR (Lead)]
              |
              +-- TeamCreate("fix-ts-errors")
              |       -> lead becomes team-lead@fix-ts-errors
              |
              +-- Analyze & decompose task into subtasks
              |       -> explore/architect produces subtask list
              |
              +-- TaskCreate x N (one per subtask)
              |       -> tasks #1, #2, #3 with dependencies
              |
              +-- TaskUpdate x N (pre-assign owners)
              |       -> task #1 owner=worker-1, etc.
              |
              +-- Task(team_name="fix-ts-errors", name="worker-1") x 3
              |       -> spawns teammates into the team
              |
              +-- Monitor loop
              |       <- SendMessage from teammates (auto-delivered)
              |       -> TaskList polling for progress
              |       -> SendMessage to unblock/coordinate
              |
              +-- Completion
                      -> SendMessage(shutdown_request) to each teammate
                      <- SendMessage(shutdown_response, approve: true)
                      -> TeamDelete("fix-ts-errors")
                      -> rm .omc/state/team-state.json

b015

Goal Workflow Relationship

b016

Team is the OMC authority for parallel, staged execution. Use the deterministic conflict policies refuse, adopt_existing, and artifact_only rather than non-deterministic warning handling. If a task mentions Claude Code /goal, Ralph, UltraQA, or artifact-only Ultragoal, keep Team as the primary loop authority unless the leader explicitly hands off. Use /goal only as a documented native Claude Code handoff target or as visible evidence from the lead session; do not claim the /goal evaluator independently runs commands, reads files, or replaces team-verify / team-fix. Artifact-only Ultragoal references should be treated as durable goal ledger/checkpoint/evidence artifacts, not as worker execution by themselves.

b017

Staged Pipeline (Canonical Team Runtime)

b018

Team execution follows a staged pipeline:

b019

team-plan -> team-prd -> team-exec -> team-verify -> team-fix (loop)

b020

Stage Agent Routing

b021

Each pipeline stage uses specialized agents -- not just executors. The lead selects agents based on the stage and task characteristics.

b022
StageRequired AgentsOptional AgentsSelection Criteria
team-planexplore (haiku), planner (opus)analyst (opus), architect (opus)Use analyst for unclear requirements. Use architect for systems with complex boundaries.
team-prdanalyst (opus)critic (opus)Use critic to challenge scope.
team-execexecutor (sonnet)executor (opus), debugger (sonnet), designer (sonnet), writer (haiku), test-engineer (sonnet)Match agent to subtask type. Use executor (model=opus) for complex autonomous work, designer for UI, debugger for compilation issues, writer for docs, test-engineer for test creation.
team-verifyverifier (sonnet)test-engineer (sonnet), security-reviewer (sonnet), code-reviewer (opus)Always run verifier. Add security-reviewer for auth/crypto changes. Add code-reviewer for >20 files or architectural changes. code-reviewer also covers style/formatting checks.
team-fixexecutor (sonnet)debugger (sonnet), executor (opus)Use debugger for type/build errors and regression isolation. Use executor (model=opus) for complex multi-file fixes.
b023

Routing rules:

b024
  1. The lead picks agents per stage, not the user. The user's N:agent-type parameter only overrides the team-exec stage worker type. All other stages use stage-appropriate specialists.
  2. Specialist agents complement executor agents. Route analysis/review to architect/critic Claude agents and UI work to designer agents. Tmux CLI workers are one-shot and don't participate in team communication.
  3. Cost mode affects model tier. In downgrade: opus agents to sonnet, sonnet to haiku where quality permits. team-verify always uses at least sonnet.
  4. Risk level escalates review. Security-sensitive or >20 file changes must include security-reviewer + code-reviewer (opus) in team-verify.
b025

Stage Entry/Exit Criteria

b026
  • team-plan
  • Entry: Team invocation is parsed and orchestration starts.
  • Agents: explore scans codebase, planner creates task graph, optionally analyst/architect for complex tasks.
  • Exit: decomposition is complete and a runnable task graph is prepared.
  • team-prd
  • Entry: scope is ambiguous or acceptance criteria are missing.
  • Agents: analyst extracts requirements, optionally critic.
  • Exit: acceptance criteria and boundaries are explicit.
  • team-exec
  • Entry: TeamCreate, TaskCreate, assignment, and worker spawn are complete.
  • Agents: workers spawned as the appropriate specialist type per subtask (see routing table).
  • Exit: execution tasks reach terminal state for the current pass.
  • team-verify
  • Entry: execution pass finishes.
  • Agents: verifier + task-appropriate reviewers (see routing table).
  • Exit (pass): verification gates pass with no required follow-up.
  • Exit (fail): fix tasks are generated and control moves to team-fix.
  • team-fix
  • Entry: verification found defects/regressions/incomplete criteria.
  • Agents: executor/debugger depending on defect type.
  • Exit: fixes are complete and flow returns to team-exec then team-verify.
b027

Verify/Fix Loop and Stop Conditions

b028

Continue team-exec -> team-verify -> team-fix until:

b029
  1. verification passes and no required fix tasks remain, or
  2. work reaches an explicit terminal blocked/failed outcome with evidence.
b030

team-fix is bounded by max attempts. If fix attempts exceed the configured limit, transition to terminal failed (no infinite loop).

b031

Stage Handoff Convention

b032

When transitioning between stages, important context — decisions made, alternatives rejected, risks identified — lives only in the lead's conversation history. If the lead's context compacts or agents restart, this knowledge is lost.

b033

Each completing stage MUST produce a handoff document before transitioning.

b034

The lead writes handoffs to .omc/handoffs/<stage-name>.md.

b035

Handoff Format

b036
## Handoff: <current-stage> → <next-stage>

- **Decided**: [key decisions made in this stage]
- **Rejected**: [alternatives considered and why they were rejected]
- **Risks**: [identified risks for the next stage]
- **Files**: [key files created or modified]
- **Remaining**: [items left for the next stage to handle]
b037

Handoff Rules

b038
  1. Lead reads previous handoff BEFORE spawning next stage's agents. The handoff content is included in the next stage's agent spawn prompts, ensuring agents start with full context.
  2. Handoffs accumulate. The verify stage can read all prior handoffs (plan → prd → exec) for full decision history.
  3. On team cancellation, handoffs survive in .omc/handoffs/ for session resume. They are not deleted by TeamDelete.
  4. Handoffs are lightweight. 10-20 lines max. They capture decisions and rationale, not full specifications (those live in deliverable files like DESIGN.md).
b039

Example

b040
## Handoff: team-plan → team-exec

- **Decided**: Microservice architecture with 3 services (auth, api, worker). PostgreSQL for persistence. JWT for auth tokens.
- **Rejected**: Monolith (scaling concerns), MongoDB (team expertise is SQL), session cookies (API-first design).
- **Risks**: Worker service needs Redis for job queue — not yet provisioned. Auth service has no rate limiting in initial design.
- **Files**: DESIGN.md, TEST_STRATEGY.md
- **Remaining**: Database migration scripts, CI/CD pipeline config, Redis provisioning.
b041

Resume and Cancel Semantics

b042
  • Resume: restart from the last non-terminal stage using staged state + live task status. Read .omc/handoffs/ to recover stage transition context.
  • Cancel: /oh-my-claudecode:cancel requests teammate shutdown, waits for responses (best effort), marks phase cancelled with active=false, captures cancellation metadata, then deletes team resources and clears/preserves Team state per policy. Handoff files in .omc/handoffs/ are preserved for potential resume.
  • Terminal states are complete, failed, and cancelled.
b043

Workflow

b044

Phase 1: Parse Input

b045
  • Extract N (agent count), validate 1-20
  • Extract agent-type, validate it maps to a known OMC subagent
  • Extract task description
b046

Phase 2: Analyze & Decompose

b047

Use explore or architect (via MCP or agent) to analyze the codebase and break the task into N subtasks:

b048
  • Each subtask should be file-scoped or module-scoped to avoid conflicts
  • Subtasks must be independent or have clear dependency ordering
  • Each subtask needs a concise subject and detailed description
  • Identify dependencies between subtasks (e.g., "shared types must be fixed before consumers")
b049

Phase 3: Create Team

b050

Call TeamCreate with a slug derived from the task:

b051
{
  "team_name": "fix-ts-errors",
  "description": "Fix all TypeScript errors across the project"
}
b052

Response:

b053
{
  "team_name": "fix-ts-errors",
  "team_file_path": "~/.claude/teams/fix-ts-errors/config.json",
  "lead_agent_id": "team-lead@fix-ts-errors"
}
b054

The current session becomes the team lead (team-lead@fix-ts-errors).

b055

Write OMC state using the state_write MCP tool for proper session-scoped persistence:

b056
state_write(mode="team", active=true, current_phase="team-plan", state={
  "team_name": "fix-ts-errors",
  "agent_count": 3,
  "agent_types": "executor",
  "task": "fix all TypeScript errors",
  "fix_loop_count": 0,
  "max_fix_loops": 3,
  "linked_ralph": false,
  "stage_history": "team-plan"
})
b057

> Note: The MCP state_write tool transports all values as strings. Consumers must coerce agent_count, fix_loop_count, max_fix_loops to numbers and linked_ralph to boolean when reading state.

b058

State schema fields:

b059

current_phasestringCurrent pipeline stage: team-plan, team-prd, team-exec, team-verify, team-fix

b060

Update state on every stage transition:

b061
state_write(mode="team", current_phase="team-exec", state={
  "stage_history": "team-plan:2026-02-07T12:00:00Z,team-prd:2026-02-07T12:01:00Z,team-exec:2026-02-07T12:02:00Z"
})
b062

Read state for resume detection:

b063
state_read(mode="team")
b064

If active=true and current_phase is non-terminal, resume from the last incomplete stage instead of creating a new team.

b069

  "activeForm": "Fixing auth type errors",

For tasks with dependencies, use TaskUpdate after creation:

b071

{
  "taskId": "3",
  "addBlockedBy": ["1"]

b073

  "taskId": "1",
  "owner": "worker-1"

Phase 5: Spawn Teammates

b075

Spawn N teammates using Task with team_name and name parameters. Each teammate gets the team worker preamble (see below) plus their specific assignment.

b076
{
  "subagent_type": "oh-my-claudecode:executor",
  "team_name": "fix-ts-errors",
  "name": "worker-1",
  "prompt": "<worker-preamble + assigned tasks>"
}
b077

Response:

b078
{
  "agent_id": "worker-1@fix-ts-errors",
  "name": "worker-1",
  "team_name": "fix-ts-errors"
}
b079

Side effects:

b080
  • Teammate added to config.json members array
  • An internal task is auto-created (with metadata._internal: true) tracking the agent lifecycle
  • Internal tasks appear in TaskList output -- filter them when counting real tasks
b081

IMPORTANT: Spawn all teammates in parallel (they are background agents). Do NOT wait for one to finish before spawning the next.

b082

Phase 6: Monitor

b083

The lead orchestrator monitors progress through two channels:

b084
  1. Inbound messages -- Teammates send SendMessage to team-lead when they complete tasks or need help. These arrive automatically as new conversation turns (no polling needed).
b085
  1. TaskList polling -- Periodically call TaskList to check overall progress:

`` #1 [completed] Fix type errors in src/auth/ (worker-1) #3 [in_progress] Fix type errors in src/api/ (worker-2) #5 [pending] Fix type errors in src/utils/ (worker-3) ` Format: #ID [status] subject (owner)`

b086

Coordination actions the lead can take:

b087
  • Unblock a teammate: Send a message with guidance or missing context
  • Reassign work: If a teammate finishes early, use TaskUpdate to assign pending tasks to them and notify via SendMessage
  • Handle failures: If a teammate reports failure, reassign the task or spawn a replacement
b088

Task Watchdog Policy

b089

Monitor for stuck or failed teammates:

b090
  • Max in-progress age: If a task stays in_progress for more than 5 minutes without messages, send a status check
  • Suspected dead worker: No messages + stuck task for 10+ minutes → reassign task to another worker
  • Reassign threshold: If a worker fails 2+ tasks, stop assigning new tasks to it
b091

Phase 6.5: Stage Transitions (State Persistence)

b092

On every stage transition, update OMC state:

b093
// Entering team-exec after planning
state_write(mode="team", current_phase="team-exec", state={
  "stage_history": "team-plan:T1,team-prd:T2,team-exec:T3"
})

// Entering team-verify after execution
state_write(mode="team", current_phase="team-verify")

// Entering team-fix after verify failure
state_write(mode="team", current_phase="team-fix", state={
  "fix_loop_count": 1
})
b094

This enables:

b095
  • Resume: If the lead crashes, state_read(mode="team") reveals the last stage and team name for recovery
  • Cancel: The cancel skill reads current_phase to know what cleanup is needed
  • Ralph integration: Ralph can read team state to know if the pipeline completed or failed
b096

Phase 7: Completion

b097

When all real tasks (non-internal) are completed or failed:

b098
  1. Verify results -- Check that all subtasks are marked completed via TaskList
  2. Shutdown teammates -- Send shutdown_request to each active teammate:

``json { "type": "shutdown_request", "recipient": "worker-1", "content": "All work complete, shutting down team" } ``

  1. Await responses -- Each teammate responds with shutdown_response(approve: true) and terminates
  2. Delete team -- Call TeamDelete to clean up:

``json { "team_name": "fix-ts-errors" } ` Response: `json { "success": true, "message": "Cleaned up directories and worktrees for team \"fix-ts-errors\"", "team_name": "fix-ts-errors" } ``

  1. Clean OMC state -- Remove .omc/state/team-state.json
  2. Report summary -- Present results to the user

Agent Preamble

b100

When spawning teammates, include this preamble in the prompt to establish the work protocol. Adapt it per teammate with their specific task assignments.

b101
You are a TEAM WORKER in team "{team_name}". Your name is "{worker_name}".
You report to the team lead ("team-lead").
You are not the leader and must not perform leader orchestration actions.

== WORK PROTOCOL ==

1. CLAIM: Call TaskList to see your assigned tasks (owner = "{worker_name}").
   Pick the first task with status "pending" that is assigned to you.
   Call TaskUpdate to set status "in_progress":
   {"taskId": "ID", "status": "in_progress", "owner": "{worker_name}"}

2. WORK: Execute the task using your tools (Read, Write, Edit, Bash).
   Do NOT spawn sub-agents. Do NOT delegate. Work directly.

3. COMPLETE: When done, mark the task completed:
   {"taskId": "ID", "status": "completed"}

4. REPORT: Notify the lead via SendMessage:
   {"type": "message", "recipient": "team-lead", "content": "Completed task #ID: <summary of what was done>", "summary": "Task #ID complete"}

5. NEXT: Check TaskList for more assigned tasks. If you have more pending tasks, go to step 1.
   If no more tasks are assigned to you, notify the lead:
   {"type": "message", "recipient": "team-lead", "content": "All assigned tasks complete. Standing by.", "summary": "All tasks done, standing by"}

6. SHUTDOWN: When you receive a shutdown_request, respond with:
   {"type": "shutdown_response", "request_id": "<from the request>", "approve": true}

== BLOCKED TASKS ==
If a task has blockedBy dependencies, skip it until those tasks are completed.
Check TaskList periodically to see if blockers have been resolved.

== ERRORS ==
If you cannot complete a task, report the failure to the lead:
{"type": "message", "recipient": "team-lead", "content": "FAILED task #ID: <reason>", "summary": "Task #ID failed"}
Do NOT mark the task as completed. Leave it in_progress so the lead can reassign.

== RULES ==
- NEVER spawn sub-agents or use the Task tool
- NEVER run tmux pane/session orchestration commands (for example `tmux split-window`, `tmux new-session`)
- NEVER run team spawning/orchestration skills or commands (for example `$team`, `$ultrawork`, `$autopilot`, `$ralph`, `omc team ...`, `omx team ...`)
- ALWAYS use absolute file paths
- ALWAYS report progress via SendMessage to "team-lead"
- Use SendMessage with type "message" only -- never "broadcast"

CLI Workers (Codex and Gemini)

b136

The team skill supports hybrid execution combining Claude agent teammates with external CLI workers (Codex CLI and Gemini CLI). Both types can make code changes -- they differ in capabilities and cost. These are standalone CLI tools, not MCP servers.

b137

Execution Modes

b138

Tasks are tagged with an execution mode during decomposition:

b139
Execution ModeProviderCapabilities
claude_workerClaude agentFull Claude Code tool access (Read/Write/Edit/Bash/Task). Best for tasks needing Claude's reasoning + iterative tool use.
codex_workerCodex CLI (tmux pane)Full filesystem access in working_directory. Runs autonomously via tmux pane. Best for code review, security analysis, refactoring, architecture. Requires npm install -g @openai/codex.
gemini_workerGemini CLI (tmux pane)Full filesystem access in working_directory. Runs autonomously via tmux pane. Best for UI/design work, documentation, large-context tasks. Requires npm install -g @google/gemini-cli.
b140

How CLI Workers Operate

b141

Tmux CLI workers run in dedicated tmux panes with filesystem access. They are autonomous executors, not just analysts:

b142
  1. Lead writes task instructions to a prompt file
  2. Lead spawns a tmux CLI worker with working_directory set to the project root
  3. The worker reads files, makes changes, runs commands -- all within the working directory
  4. Results/summary are written to an output file
  5. Lead reads the output, marks the task complete, and feeds results to dependent tasks
b143

Key difference from Claude teammates:

b144
  • CLI workers operate via tmux, not Claude Code's tool system
  • They cannot use TaskList/TaskUpdate/SendMessage (no team awareness)
  • They run as one-shot autonomous jobs, not persistent teammates
  • The lead manages their lifecycle (spawn, monitor, collect results)
b145

Example: Hybrid Team with CLI Workers

b148
/team 3:executor "refactor auth module with security review"

Task decomposition:
#1 [codex_worker] Security review of current auth code -> output to .omc/research/auth-security.md
#2 [codex_worker] Refactor auth/login.ts and auth/session.ts (uses #1 findings)
#3 [claude_worker:designer] Redesign auth UI components (login form, session indicator)
#4 [claude_worker] Update auth tests + fix integration issues

b149

The lead runs #1 (Codex security analysis), then #2 and #3 in parallel (Codex refactors backend, designer agent redesigns frontend), then #4 (Claude teammate handles test iteration), then #5 (Gemini final review).

Monitor Enhancement: Outbox Auto-Ingestion

b155

The lead can proactively ingest outbox messages from CLI workers using the outbox reader utilities, enabling event-driven monitoring without relying solely on SendMessage delivery.

b156

Outbox Reader Functions

b157

readNewOutboxMessages(teamName, workerName) -- Read new outbox messages for a single worker using a byte-offset cursor. Each call advances the cursor, so subsequent calls only return messages written since the last read. Mirrors the inbox cursor pattern from readNewInboxMessages().

b158

readAllTeamOutboxMessages(teamName) -- Read new outbox messages from ALL workers in a team. Returns an array of { workerName, messages } entries, skipping workers with no new messages. Useful for batch polling in the monitor loop.

b159

resetOutboxCursor(teamName, workerName) -- Reset the outbox cursor for a worker back to byte 0. Useful when re-reading historical messages after a lead restart or for debugging.

Using getTeamStatus() in the Monitor Phase

b161

The getTeamStatus(teamName, workingDirectory, heartbeatMaxAgeMs?) function provides a unified snapshot combining:

b162
  • Worker registration -- Which MCP workers are registered (from shadow registry / config.json)
  • Heartbeat freshness -- Whether each worker is alive based on heartbeat age
  • Task progress -- Per-worker and team-wide task counts (pending, in_progress, completed)
  • Current task -- Which task each worker is actively executing
  • Recent outbox messages -- New messages since the last status check
b163

Example usage in the monitor loop:

b164
const status = getTeamStatus("fix-ts-errors", workingDirectory);

for (const worker of status.workers) {
  if (!worker.isAlive) {
    // Worker is dead -- reassign its in-progress tasks
  }
  for (const msg of worker.recentMessages) {
    if (msg.type === "task_complete") {
      // Mark task complete, unblock dependents
    } else if (msg.type === "task_failed") {
      // Handle failure, possibly retry or reassign
    } else if (msg.type === "error") {
      // Log error, check if worker needs intervention
    }
  }
}

if (status.taskSummary.pending === 0 && status.taskSummary.inProgress === 0) {
  // All work done -- proceed to shutdown
}

Error Handling

b169

Teammate Fails a Task

b170
  1. Teammate sends SendMessage to lead reporting the failure
  2. Lead decides: retry (reassign same task to same or different worker) or skip
  3. To reassign: TaskUpdate to set new owner, then SendMessage to the new owner
b171

Teammate Gets Stuck (No Messages)

b172
  1. Lead detects via TaskList -- task stuck in in_progress for too long
  2. Lead sends SendMessage to the teammate asking for status
  3. If no response, consider the teammate dead
  4. Reassign the task to another worker via TaskUpdate
b173

Dependency Blocked

b174
  1. If a blocking task fails, the lead must decide whether to:
  • Retry the blocker
  • Remove the dependency (TaskUpdate with modified blockedBy)
  • Skip the blocked task entirely
  1. Communicate decisions to affected teammates via SendMessage
b175

Teammate Crashes

b176
  1. Internal task for that teammate will show unexpected status
  2. Teammate disappears from config.json members
  3. Lead reassigns orphaned tasks to remaining workers
  4. If needed, spawn a replacement teammate with Task(team_name, name)

Cancellation

b200

The /oh-my-claudecode:cancel skill handles team cleanup:

b201
  1. Read team state via state_read(mode="team") to get team_name and linked_ralph
  2. Send shutdown_request to all active teammates (from config.json members)
  3. Wait for shutdown_response from each (15s timeout per member)
  4. Call TeamDelete to remove team and task directories
  5. Clear state via state_clear(mode="team")
  6. If linked_ralph is true, also clear ralph: state_clear(mode="ralph")
b202

Linked Mode Cancellation (Team + Ralph)

b203

When team is linked to ralph, cancellation follows dependency order:

b204
  • Cancel triggered from Ralph context: Cancel Team first (graceful shutdown of all teammates), then clear Ralph state. This ensures workers are stopped before the persistence loop exits.
  • Cancel triggered from Team context: Clear Team state, then mark Ralph as cancelled. Ralph's stop hook will detect the missing team and stop iterating.
  • Force cancel (--force): Clears both team and ralph state unconditionally via state_clear.
b205

If teammates are unresponsive, TeamDelete may fail. In that case, the cancel skill should wait briefly and retry, or inform the user to manually clean up ~/.claude/teams/{team_name}/ and ~/.claude/tasks/{team_name}/.

Runtime V2 (Event-Driven)

b207

When OMC_RUNTIME_V2=1 is set, the team runtime uses an event-driven architecture instead of the legacy done.json polling watchdog:

b208
  • No done.json: Task completion is detected via CLI API lifecycle transitions (claim-task, transition-task-status)
  • Snapshot-based monitoring: Each poll cycle takes a point-in-time snapshot of tasks and workers, computes deltas, and emits events
  • Event log: All team events are appended to .omc/state/team/{teamName}/events.jsonl
  • Worker status files: Workers write status to .omc/state/team/{teamName}/workers/{name}/status.json
  • Preserved: Sentinel gate (blocks premature completion), circuit breaker (dead worker detection), failure sidecars
b209

The v2 runtime is feature-flagged and can be enabled per-session. The legacy v1 runtime remains the default.

Dynamic Scaling

b211

Git Worktree Integration

b244

MCP workers can operate in isolated git worktrees to prevent file conflicts between concurrent workers.

b245

How It Works

b246
  1. Worktree creation: Before spawning a worker, call createWorkerWorktree(teamName, workerName, repoRoot) to create an isolated worktree at .omc/worktrees/{team}/{worker} with branch omc-team/{teamName}/{workerName}.
b247
  1. Worker isolation: Pass the worktree path as the workingDirectory in the worker's BridgeConfig. The worker operates exclusively in its own worktree.
b248
  1. Merge coordination: After a worker completes its tasks, use checkMergeConflicts() to verify the branch can be cleanly merged, then mergeWorkerBranch() to merge with --no-ff for clear history.
b249
  1. Team cleanup: On team shutdown, call cleanupTeamWorktrees(teamName, repoRoot) to remove all worktrees and their branches.
b250

b254

Gotchas

b255
  1. Internal tasks pollute TaskList -- When a teammate is spawned, the system auto-creates an internal task with metadata._internal: true. These appear in TaskList output. Filter them when counting real task progress. The subject of an internal task is the teammate's name.
b256
  1. No atomic claiming -- Unlike SQLite swarm, there is no transactional guarantee on TaskUpdate. Two teammates could race to claim the same task. Mitigation: The lead should pre-assign owners via TaskUpdate(taskId, owner) before spawning teammates. Teammates should only work on tasks assigned to them.
b257
  1. Task IDs are strings -- IDs are auto-incrementing strings ("1", "2", "3"), not integers. Always pass string values to taskId fields.
b258
  1. TeamDelete requires empty team -- All teammates must be shut down before calling TeamDelete. The lead (the only remaining member) is excluded from this check.
b259
  1. Messages are auto-delivered -- Teammate messages arrive to the lead as new conversation turns. No polling or inbox-checking is needed for inbound messages. However, if the lead is mid-turn (processing), messages queue and deliver when the turn ends.
b260
  1. Teammate prompt stored in config -- The full prompt text is stored in config.json members array. Do not put secrets or sensitive data in teammate prompts.
b261
  1. Members auto-removed on shutdown -- After a teammate approves shutdown and terminates, it is automatically removed from config.json. Do not re-read config expecting to find shut-down teammates.
b262
  1. shutdown_response needs request_id -- The teammate must extract the request_id from the incoming shutdown request JSON and pass it back. The format is shutdown-{timestamp}@{worker-name}. Fabricating this ID will cause the shutdown to fail silently.
b263
  1. Team name must be a valid slug -- Use lowercase letters, numbers, and hyphens. Derive from the task description (e.g., "fix TypeScript errors" becomes "fix-ts-errors").
b264
  1. Broadcast is expensive -- Each broadcast sends a separate message to every teammate. Use message (DM) by default. Only broadcast for truly team-wide critical alerts.
b265
  1. CLI workers are one-shot, not persistent -- Tmux CLI workers have full filesystem access and CAN make code changes. However, they run as autonomous one-shot jobs -- they cannot use TaskList/TaskUpdate/SendMessage. The lead must manage their lifecycle: write prompt_file, spawn CLI worker, read output_file, mark task complete. They don't participate in team communication like Claude teammates do.

b023

路由規則：

b024
  1. 負責人依每個階段選擇代理程式，而不是使用者。 使用者的 N:agent-type 參數只會覆寫 team-exec 階段的工作者類型。所有其他階段都使用適合該階段的專家。
  2. 專家代理程式補充執行代理程式。 將分析/審查路由給 architect/critic Claude 代理程式，將 UI 工作路由給 designer 代理程式。Tmux CLI 工作者是一次性的，不參與團隊溝通。
  3. 成本模式會影響模型層級。 在降級時：在品質允許的情況下，將 opus 代理程式降為 sonnet，將 sonnet 降為 haiku。team-verify 一律至少使用 sonnet。
  4. 風險等級會升級審查。 安全敏感變更或超過 20 個檔案的變更，必須在 team-verify 中包含 security-reviewer + code-reviewer（opus）。
b025

階段進入/退出條件

b026
  • team-plan
  • 進入：團隊呼叫被解析，協調開始。
  • 代理程式：explore 掃描程式碼庫，planner 建立任務圖，複雜任務可選擇 analyst/architect。
  • 退出：分解完成，且已準備好可執行的任務圖。
  • team-prd
  • 進入：範圍不明確或缺少驗收標準。
  • 代理程式：analyst 擷取需求，必要時加入 critic。
  • 退出：驗收標準與邊界已明確。
  • team-exec
  • 進入：TeamCreate、TaskCreate、指派與工作者啟動已完成。
  • 代理程式：依每個子任務啟動對應的專家類型工作者（見路由表）。
  • 退出：執行任務在目前輪次達到終止狀態。
  • team-verify
  • 進入：執行輪次結束。
  • 代理程式：verifier + 適合該任務的審查者（見路由表）。
  • 退出（通過）：驗證閘道通過，且沒有必須跟進的事項。
  • 退出（失敗）：產生修正任務，控制權轉移到 team-fix。
  • team-fix
  • 進入：驗證發現缺陷/回歸/不完整標準。
  • 代理程式：依缺陷類型使用 executor/debugger。
  • 退出：修正完成，流程回到 team-exec，然後 team-verify。
b027

驗證/修正迴圈與停止條件

b028

持續 team-exec -> team-verify -> team-fix，直到：

b029
  1. 驗證通過且沒有剩餘的必要修正任務，或
  2. 工作到達帶有證據的明確終止 blocked/failed 結果。
b030

team-fix 受最大嘗試次數限制。如果修正嘗試超過設定上限，則轉移到終止狀態 failed（不會無限迴圈）。

b031

階段交接慣例

b032

在階段之間轉換時，重要脈絡——已做出的決策、被拒絕的替代方案、已識別的風險——只存在於負責人的對話歷史中。如果負責人的上下文被壓縮或代理程式重新啟動，這些知識就會遺失。

b033

每個完成的階段在轉換前都必須產生一份交接文件。

b034

負責人將交接內容寫入 .omc/handoffs/<stage-name>.md。

b035

交接格式

b036
## Handoff: <current-stage> → <next-stage>

- **Decided**: [key decisions made in this stage]
- **Rejected**: [alternatives considered and why they were rejected]
- **Risks**: [identified risks for the next stage]
- **Files**: [key files created or modified]
- **Remaining**: [items left for the next stage to handle]
b037

交接規則

b038
  1. 負責人在啟動下一階段的代理程式之前，必須先讀取前一份交接。 交接內容會包含在下一階段的代理程式啟動提示中，確保代理程式從完整脈絡開始。
  2. 交接會累積。 驗證階段可以讀取所有先前的交接（plan → prd → exec），以取得完整的決策歷史。
  3. 在團隊取消時，交接會保留 在 .omc/handoffs/ 中供工作階段恢復使用。它們不會被 TeamDelete 刪除。
  4. 交接應保持輕量。 最多 10-20 行。它們記錄決策與理由，而不是完整規格（那些內容會放在像 DESIGN.md 這類交付檔案中）。
b039

範例

b040
## 交接：team-plan → team-exec

- **已決定**：採用包含 3 個服務（auth、api、worker）的微服務架構。使用 PostgreSQL 作為持久化儲存。使用 JWT 作為 auth token。
- **已拒絕**：單體架構（擴展性疑慮）、MongoDB（團隊專長是 SQL）、session cookies（API-first 設計）。
- **風險**：Worker 服務需要 Redis 作為 job queue —— 目前尚未配置。Auth 服務在初始設計中沒有 rate limiting。
- **檔案**：DESIGN.md, TEST_STRATEGY.md
- **剩餘事項**：資料庫 migration scripts、CI/CD pipeline config、Redis provisioning。
b041

接續與取消語義

b042
  • 接續： 使用分階段狀態 + 即時任務狀態，從最後一個非終止階段重新開始。讀取 .omc/handoffs/ 以還原階段轉換脈絡。
  • 取消： /oh-my-claudecode:cancel 會請求隊友關閉，等待回應（盡力而為），將階段標記為 cancelled 並設為 active=false，擷取取消中繼資料，然後刪除團隊資源，並依政策清除/保留 Team 狀態。.omc/handoffs/ 中的交接檔會保留，以便日後可能接續。
  • 終止狀態是 complete、failed 與 cancelled。
b043

工作流程

b044

階段 1：解析輸入

b045
  • 擷取 N（代理數量），驗證範圍為 1-20
  • 擷取 agent-type，驗證其可對應到已知的 OMC 子代理
  • 擷取 task 描述
b046

階段 2：分析與拆解

b047

使用 explore 或 architect（透過 MCP 或 agent）分析程式碼庫，並將任務拆分為 N 個子任務：

b048
  • 每個子任務都應該是 file-scoped 或 module-scoped，以避免衝突
  • 子任務必須彼此獨立，或具備清楚的相依順序
  • 每個子任務都需要簡潔的 subject 與詳細的 description
  • 識別子任務之間的相依性（例如："shared types must be fixed before consumers"）
b049

階段 3：建立團隊

b050

使用從任務衍生的 slug 呼叫 TeamCreate：

b051
{
  "team_name": "fix-ts-errors",
  "description": "修正整個專案中的所有 TypeScript 錯誤"
}
b052

回應：

b053
{
  "team_name": "fix-ts-errors",
  "team_file_path": "~/.claude/teams/fix-ts-errors/config.json",
  "lead_agent_id": "team-lead@fix-ts-errors"
}
b054

目前 session 會成為 team lead（team-lead@fix-ts-errors）。

b055

使用 state_write MCP 工具寫入 OMC 狀態，以便進行正確的 session 範圍持久化：

b056
state_write(mode="team", active=true, current_phase="team-plan", state={
  "team_name": "fix-ts-errors",
  "agent_count": 3,
  "agent_types": "executor",
  "task": "修正所有 TypeScript 錯誤",
  "fix_loop_count": 0,

> 注意： MCP state_write 工具會以字串傳遞所有值。讀取狀態時，消費端必須將 agent_count、fix_loop_count、max_fix_loops 轉成數字，並將 linked_ralph 轉成布林值。

b058

狀態結構欄位：

b059
欄位型別說明

  "stage_history": "team-plan:2026-02-07T12:00:00Z,team-prd:2026-02-07T12:01:00Z,team-exec:2026-02-07T12:02:00Z"
})
b062

為了偵測接續而讀取狀態：

b063
state_read(mode="team")
b064

如果 active=true 且 current_phase 為非終止狀態，則從最後一個未完成的階段接續，而不是建立新團隊。

b075

使用帶有 team_name 和 name 參數的 Task 產生 N 位隊友。每位隊友都會獲得團隊 worker 前言（見下文）以及其專屬指派。

b076
{
  "subagent_type": "oh-my-claudecode:executor",
  "team_name": "fix-ts-errors",

  "prompt": "<worker-preamble + assigned tasks>"
}

回應：

b078
{
  "agent_id": "worker-1@fix-ts-errors",
  "name": "worker-1",
  "team_name": "fix-ts-errors"
}
b079

副作用：

b080
  • 新增隊友至 config.json 的 members 陣列
  • 會自動建立一個 internal task（metadata._internal: true），用來追蹤代理生命週期
  • Internal tasks 會出現在 TaskList 輸出中 -- 在計算實際任務時請將它們過濾掉
b081

重要： 請平行啟動所有隊友（它們是背景代理）。在啟動下一個之前，不要等待某一個完成。

b082

第 6 階段：監控

b083

主協調者透過兩個管道監控進度：

b084
  1. 傳入訊息 -- 隊友在完成任務或需要協助時，會向 team-lead 傳送 SendMessage。這些訊息會自動以新的對話回合送達（不需要輪詢）。
b085
  1. TaskList 輪詢 -- 定期呼叫 TaskList 以檢查整體進度：

`` #1 [completed] Fix type errors in src/auth/ (worker-1) #3 [in_progress] Fix type errors in src/api/ (worker-2) #5 [pending] Fix type errors in src/utils/ (worker-3) ` 格式：#ID [status] subject (owner)`

b086

主導者可以執行的協調動作：

b087
  • 解除隊友阻塞： 傳送一則附帶指引或缺失上下文的 message
  • 重新分配工作： 如果隊友提早完成，使用 TaskUpdate 將待處理任務指派給他們，並透過 SendMessage 通知
  • 處理失敗： 如果隊友回報失敗，重新分配任務或啟動替代者
b088

任務看門狗政策

b089

監控卡住或失敗的隊友：

b090
  • 最長進行中時間：如果某個任務在沒有訊息的情況下維持 in_progress 超過 5 分鐘，請送出狀態確認
  • 疑似失聯的 worker：沒有訊息 + 任務卡住超過 10 分鐘 → 將任務重新指派給其他 worker
  • 重新指派門檻：如果某個 worker 失敗 2 個以上任務，就停止再指派新任務給它
b091

第 6.5 階段：階段轉換（狀態持久化）

b092

每次階段轉換時，都要更新 OMC 狀態：

b093
// 在規劃後進入 team-exec
state_write(mode="team", current_phase="team-exec", state={
  "stage_history": "team-plan:T1,team-prd:T2,team-exec:T3"
})

// 在執行後進入 team-verify
state_write(mode="team", current_phase="team-verify")

// 在驗證失敗後進入 team-fix
state_write(mode="team", current_phase="team-fix", state={
  "fix_loop_count": 1
})
b094

這使得以下成為可能：

b095
  • 續跑：如果主導者當機，state_read(mode="team") 會揭示最後階段與隊伍名稱，以便復原
  • 取消：cancel skill 會讀取 current_phase 以了解需要哪些清理工作
  • Ralph 整合：Ralph 可以讀取 team state，以判斷 pipeline 是完成還是失敗
b096

第 7 階段：完成

b097

當所有實際任務（非 internal）都已完成或失敗時：

b098
  1. 驗證結果 -- 透過 TaskList 檢查所有子任務是否都已標記為 completed
  2. 關閉隊友 -- 向每個啟用中的隊友傳送 shutdown_request：

``json { "type": "shutdown_request", "recipient": "worker-1", "content": "所有工作已完成，正在關閉團隊" } ``

  1. 等待回應 -- 每個隊友都會以 shutdown_response(approve: true) 回應並終止
  2. 刪除隊伍 -- 呼叫 TeamDelete 進行清理：

``json { "team_name": "fix-ts-errors" } ` 回應： `json { "success": true, "message": "已清理團隊 \"fix-ts-errors\" 的目錄與工作樹", "team_name": "fix-ts-errors" } ``

  1. 清理 OMC 狀態 -- 移除 .omc/state/team-state.json
  2. 回報摘要 -- 向使用者呈現結果

代理前言

b100

在啟動隊友時，請將這段前言加入提示中，以建立工作協定。請依照每位隊友的具體任務分配進行調整。

b101
你是 team "{team_name}" 中的一名 TEAM WORKER。你的名稱是 "{worker_name}"。
你向 team lead（"team-lead"）回報。
你不是 leader，且不得執行 leader 的協調動作。

== 工作協定 ==

1. CLAIM：呼叫 TaskList 查看你被指派的任務（owner = "{worker_name}"）。
   選擇第一個狀態為 "pending" 且指派給你的任務。
   呼叫 TaskUpdate 將狀態設為 "in_progress"：
   {"taskId": "ID", "status": "in_progress", "owner": "{worker_name}"}

2. WORK：使用你的工具（Read、Write、Edit、Bash）執行任務。
   不要啟動子代理。不要委派。直接工作。

3. COMPLETE：完成後，將任務標記為 completed：
   {"taskId": "ID", "status": "completed"}

4. REPORT：透過 SendMessage 通知 lead：
   {"type": "message", "recipient": "team-lead", "content": "Completed task #ID: <summary of what was done>", "summary": "Task #ID complete"}

5. NEXT：檢查 TaskList 是否還有其他已指派任務。如果你還有更多 pending 任務，回到步驟 1。
   如果沒有更多任務指派給你，通知 lead：
   {"type": "message", "recipient": "team-lead", "content": "All assigned tasks complete. Standing by.", "summary": "All tasks done, standing by"}

6. SHUTDOWN：當你收到 shutdown_request 時，請以以下內容回應：
   {"type": "shutdown_response", "request_id": "<from the request>", "approve": true}

== 被阻塞的任務 ==
如果某個任務具有 blockedBy 依賴關係，請先跳過，直到那些任務完成為止。
定期檢查 TaskList 以確認阻塞條件是否已解除。

== 錯誤 ==
如果你無法完成某個任務，請向 lead 回報失敗：
{"type": "message", "recipient": "team-lead", "content": "FAILED task #ID: <reason>", "summary": "Task #ID failed"}
不要將任務標記為 completed。讓它保持 in_progress，方便 lead 重新指派。

== 規則 ==
- 絕不啟動子代理或使用 Task 工具
- 絕不執行 tmux 面板/工作階段協調命令（例如 `tmux split-window`、`tmux new-session`）
- 絕不執行 team 啟動/協調的 skills 或命令（例如 `$team`、`$ultrawork`、`$autopilot`、`$ralph`、`omc team ...`、`omx team ...`）
- 一律使用絕對檔案路徑
- 一律透過 SendMessage 向 "team-lead" 回報進度
- 只使用 type 為 "message" 的 SendMessage -- 絕不使用 "broadcast"

關機程序（阻斷性）

b114

重要：步驟必須依照精確順序執行。在確認 shutdown 之前，絕不可呼叫 TeamDelete。

b115

步驟 1：確認完成

b116
呼叫 TaskList — 驗證所有實際任務（非內部任務）都已完成或失敗。
b117

步驟 2：向每位隊友請求關機

b118

領導者傳送：

b119
{
  "type": "shutdown_request",
  "recipient": "worker-1",
  "content": "所有工作已完成，正在關閉團隊"
}
b120

步驟 3：等待回應（阻斷性）

b121
  • 每位隊友最多等待 30 秒以取得 shutdown_response
  • 追蹤哪些隊友已確認、哪些已逾時
  • 如果某位隊友在 30 秒內沒有回應：記錄警告，標記為無回應
b122

隊友接收並回應：

b123
{
  "type": "shutdown_response",
  "request_id": "shutdown-1770428632375@worker-1",
  "approve": true
}
b124

核准後：

b125
  • 隊友程序終止
  • 該隊友會自動從 config.json 的 members 陣列中移除
  • 該隊友的內部任務完成
b126

步驟 4：TeamDelete — 只有在所有隊友都已確認或逾時之後

b127
{ "team_name": "fix-ts-errors" }
b128

步驟 5：孤兒掃描

b129

檢查在 TeamDelete 後仍存活的代理程序：

b130
node "${CLAUDE_PLUGIN_ROOT}/scripts/cleanup-orphans.mjs" --team-name fix-ts-errors
b131

此指令會掃描與團隊名稱相符、但其設定已不再存在的程序，並將其終止（SIGTERM → 等待 5 秒 → SIGKILL）。支援 --dry-run 供檢視。

b132

關機序列具有阻斷性： 在所有隊友都已具備以下任一情況之前，不要繼續執行 TeamDelete：

b133
  • 已確認關機（shutdown_response 且 approve: true），或
  • 已逾時（30 秒內無回應）
b134

重要： request_id 會提供在隊友收到的關機請求訊息中。隊友必須將其擷取並回傳。請勿捏造 request ID。

CLI 工作者如何運作

b141

Tmux CLI 工作者在具有檔案系統存取權的專用 tmux pane 中執行。它們是自主執行者，不只是分析員：

b142
  1. 領導者將任務指示寫入提示檔案
  2. 領導者啟動一個 tmux CLI 工作者，並將 working_directory 設為專案根目錄
  3. 工作者讀取檔案、進行變更、執行指令 -- 全都在工作目錄內完成
  4. 結果／摘要會寫入輸出檔案
  5. 領導者讀取輸出，將任務標記為完成，並把結果提供給相依任務

b144
  • CLI 工作者透過 tmux 運作，而不是透過 Claude Code 的工具系統
  • 它們不能使用 TaskList/TaskUpdate/SendMessage（沒有團隊感知）
  • 它們以一次性自主工作執行，而不是持續性的隊友
  • 領導者管理它們的生命週期（啟動、監控、收集結果）

監控增強：Outbox 自動擷取

b155

lead 可以使用 outbox reader 工具主動擷取 CLI worker 的 outbox 訊息，實現事件驅動監控，而不必完全依賴 SendMessage 傳遞。

b156

Outbox Reader 函式

b157

readNewOutboxMessages(teamName, workerName) -- 使用位元組偏移游標讀取單一 worker 的新 outbox 訊息。每次呼叫都會前進游標，因此後續呼叫只會回傳自上次讀取以來寫入的訊息。與 readNewInboxMessages() 的 inbox 游標模式相同。

b158

readAllTeamOutboxMessages(teamName) -- 讀取團隊中所有 worker 的新 outbox 訊息。回傳一個由 { workerName, messages } 項目組成的陣列，會略過沒有新訊息的 worker。適合在監控迴圈中進行批次輪詢。

b159

resetOutboxCursor(teamName, workerName) -- 將某個 worker 的 outbox 游標重設回位元組 0。適合在 lead 重新啟動後重新讀取歷史訊息，或用於除錯。

b160

在監控階段使用 getTeamStatus()

b161

getTeamStatus(teamName, workingDirectory, heartbeatMaxAgeMs?) 函式提供一個統一快照，整合：

b162
  • Worker 註冊 -- 哪些 MCP worker 已註冊（來自 shadow registry / config.json）
  • Heartbeat 新鮮度 -- 依據 heartbeat 的時間判斷每個 worker 是否存活
  • 任務進度 -- 每個 worker 與整個團隊的任務數量（pending、in_progress、completed）
  • 目前任務 -- 每個 worker 正在執行哪個任務
  • 最近的 outbox 訊息 -- 自上次狀態檢查以來的新訊息
b163

監控迴圈中的範例用法：

b164
const status = getTeamStatus("fix-ts-errors", workingDirectory);

for (const worker of status.workers) {
  if (!worker.isAlive) {
    // Worker 已死亡 -- 重新指派其進行中的任務
  }
  for (const msg of worker.recentMessages) {
    if (msg.type === "task_complete") {
      // 將任務標記為完成，解除依賴者的阻塞
    } else if (msg.type === "task_failed") {
      // 處理失敗，可能重試或重新指派
    } else if (msg.type === "error") {
      // 記錄錯誤，檢查 worker 是否需要介入
    }
  }
}

if (status.taskSummary.pending === 0 && status.taskSummary.inProgress === 0) {
  // 所有工作已完成 -- 進入關閉程序
}
b165

由 Outbox 訊息觸發的事件式動作

b166
訊息類型動作
task_complete將任務標記為已完成，檢查被阻塞的任務是否已解除阻塞，通知相依的 worker
task_failed增加失敗計數器，決定重試、重新指派或略過
idleWorker 沒有被指派的任務 -- 指派待處理工作或開始關閉程序
error記錄錯誤，檢查 heartbeat 中的 consecutiveErrors 是否達到隔離門檻
shutdown_ackWorker 已確認關閉 -- 可安全地從團隊移除
heartbeat更新存活追蹤（與 heartbeat 檔案重複，但對延遲監控有用）
b167

這種方法為無法使用 Claude Code 團隊訊息工具的 MCP worker 提供一種 pull-based 機制，從而補強現有的、基於 SendMessage 的通訊。

b168

錯誤處理

b169

隊友任務失敗

b170
  1. 隊友透過 SendMessage 向 lead 回報失敗
  2. lead 決定：重試（將相同任務重新指派給同一個或不同的 worker）或略過
  3. 若要重新指派：先用 TaskUpdate 設定新的擁有者，再用 SendMessage 通知新的擁有者
b171

隊友卡住了（沒有訊息）

b172
  1. lead 透過 TaskList 偵測到 -- 任務在 in_progress 狀態停留太久
  2. lead 向隊友傳送 SendMessage 詢問狀態
  3. 如果沒有回應，就將該隊友視為已死亡
  4. 透過 TaskUpdate 將任務重新指派給另一個 worker
b173

依賴被阻塞

b174
  1. 如果某個阻塞任務失敗，lead 必須決定是否要：
  • 重試阻塞者
  • 移除該依賴（使用修改過 blockedBy 的 TaskUpdate）
  • 直接略過被阻塞的任務
  1. 透過 SendMessage 將決策通知受影響的隊友
b175

隊友當機

b176
  1. 該隊友的內部任務會顯示非預期的狀態
  2. 隊友會從 config.json 的 members 中消失
  3. lead 將孤兒任務重新指派給剩餘的 worker
  4. 如有需要，使用 Task(team_name, name) 啟動一個替代隊友
b177

Team + Ralph 組合

b178

當使用者呼叫 /team ralph、說出「team ralph」，或同時組合這兩個關鍵字時，team 模式會包裹在 Ralph 的持續執行迴圈中。這會提供：

b179
  • Team 協調 -- 具備每個階段專門代理的多代理分階段 pipeline
  • Ralph 持續性 -- 失敗時重試、完成前進行架構驗證、迭代追蹤
b180

啟用

b181

Team+Ralph 會在以下情況啟用：

b182
  1. 使用者呼叫 /team ralph "task" 或 /oh-my-claudecode:team ralph "task"
  2. 關鍵字偵測器在提示中同時找到 team 與 ralph
  3. hook 在團隊上下文中偵測到 MAGIC KEYWORD: RALPH
b183

狀態連結

b184

兩種模式都會寫入各自的狀態檔案，並互相交叉引用：

b185
// 團隊狀態（透過 state_write）
state_write(mode="team", active=true, current_phase="team-plan", state={
  "team_name": "build-rest-api",
  "linked_ralph": true,
  "task": "建置完整的 REST API"
})

// Ralph 狀態（透過 state_write）
state_write(mode="ralph", active=true, iteration=1, max_iterations=10, current_phase="execution", state={
  "linked_team": true,
  "team_name": "build-rest-api"
})
b186

執行流程

b187
  1. Ralph 外層迴圈開始（第 1 次迭代）
  2. 團隊流水線執行：team-plan -> team-prd -> team-exec -> team-verify
  3. 如果 team-verify 通過：Ralph 執行架構師驗證（至少 STANDARD 層級）
  4. 如果架構師核准：兩種模式都完成，執行 /oh-my-claudecode:cancel
  5. 如果 team-verify 失敗，或架構師拒絕：團隊進入 team-fix，然後迴圈回到 team-exec -> team-verify
  6. 如果修正迴圈超過 max_fix_loops：Ralph 遞增迭代次數並重試整個流水線
  7. 如果 Ralph 超過 max_iterations：進入終止 failed 狀態
b188

取消

b189

取消任一模式都會同時取消兩者：

b190
  • 取消 Ralph（已連結）: 先取消 Team（優雅關閉），再清除 Ralph 狀態
  • 取消 Team（已連結）: 清除 Team，將 Ralph 迭代標記為已取消，停止迴圈

詳情請見下方的取消章節。

冪等復原

b193

如果 lead 在執行途中當機，team skill 應該偵測既有狀態並恢復執行：

  1. 檢查 ${CLAUDE_CONFIG_DIR:-~/.claude}/teams/ 中是否有與任務 slug 相符的團隊

取消

b200

/oh-my-claudecode:cancel skill 負責處理 team 清理：

b201
  1. 透過 state_read(mode="team") 讀取 team 狀態，以取得 team_name 和 linked_ralph
  2. 向所有作用中的隊友傳送 shutdown_request（從 config.json 的成員取得）
  3. 等待每個成員回傳 shutdown_response（每位成員 15 秒逾時）
  4. 呼叫 TeamDelete 移除 team 與 task 目錄
  5. 透過 state_clear(mode="team") 清除狀態
  6. 如果 linked_ralph 為 true，也一併清除 ralph：state_clear(mode="ralph")
b202

連結模式取消（Team + Ralph）

b203

當 team 連結到 ralph 時，取消會依照相依順序進行：

b204
  • 從 Ralph 內容觸發取消： 先取消 Team（讓所有隊友優雅關閉），再清除 Ralph 狀態。這可確保在持久化迴圈結束前，工作者已停止。
  • 從 Team 內容觸發取消： 清除 Team 狀態，然後將 Ralph 標記為已取消。Ralph 的停止 hook 會偵測到缺失的 team，並停止迭代。
  • 強制取消（--force）： 透過 state_clear 無條件清除 team 與 ralph 兩種狀態。
b205

如果隊友沒有回應，TeamDelete 可能會失敗。在這種情況下，cancel skill 應該稍等片刻後重試，或通知使用者手動清理 ~/.claude/teams/{team_name}/ 與 ~/.claude/tasks/{team_name}/。

Runtime V2（事件驅動）

b207

當設定 OMC_RUNTIME_V2=1 時，team runtime 會使用事件驅動架構，而不是舊版的 done.json 輪詢看門狗：

b208
  • 沒有 done.json：任務完成是透過 CLI API 生命週期轉換偵測（claim-task、transition-task-status）
  • 基於快照的監控：每個輪詢週期都會取得任務與工作者的即時快照，計算差異，並發出事件
  • 事件記錄：所有 team 事件都會附加寫入 .omc/state/team/{teamName}/events.jsonl
  • 工作者狀態檔：工作者將狀態寫入 .omc/state/team/{teamName}/workers/{name}/status.json
  • 保留：sentinel gate（阻止過早完成）、circuit breaker（偵測死掉的工作者）、failure sidecars
b209

v2 runtime 受到功能旗標控制，且可按每個 session 啟用。舊版 v1 runtime 仍然是預設值。

b210

動態擴縮

b211

當設定 OMC_TEAM_SCALING_ENABLED=1 時，team 支援在 session 進行中擴縮：

b212
  • scale_up：向正在執行的團隊新增工作者（遵守 max_workers 上限）
  • scale_down：以優雅排空的方式移除閒置工作者（工作者會在移除前完成目前的任務）
  • 基於檔案的擴縮鎖可防止並行的擴縮操作
  • 單調遞增的工作者索引計數器可確保跨擴縮事件時工作者名稱唯一

b214

選用設定位於 .claude/omc.jsonc（專案）或 ~/.config/claude-omc/config.jsonc（使用者）。專案值會覆蓋使用者值；OMC_TEAM_ROLE_OVERRIDES（環境變數 JSON）會優先於兩者。

b215
{
  "team": {
    "ops": {
      "maxAgents": 20,
      "defaultAgentType": "claude",

  • ops.maxAgents - 最大隊友數（預設：20）
  • ops.defaultAgentType - 當 /team 呼叫未指定時使用的 CLI 提供者（claude | codex | gemini，預設：claude）
  • ops.monitorIntervalMs - TaskList 的輪詢頻率（預設：30 秒）
  • ops.shutdownTimeoutMs - 等待關機回應的時間長度（預設：15 秒）
b217

> 注意： 團隊成員沒有硬編碼的模型預設值。每個隊友都是一個獨立的 Claude Code 工作階段，會繼承使用者設定的模型。由於隊友可以自行生成子代理，工作階段模型扮演編排層，而子代理可以使用任何模型階級。

b218

每個角色的提供者與模型路由

b219

> 範圍： 僅適用於 /team。以任務為基礎的委派使用 delegationRouting（請參見其他文件）。這兩個系統是刻意共存的。

b220

宣告每個標準角色應由哪個提供者（claude、codex、gemini）以及哪個模型階級支援。路由會在團隊建立時只解析一次，並持久化到 TeamConfig.resolved_routing 中——spawn、scale-up 與 restart 都會從這個快照讀取，因此某個角色的工作者 CLI 與模型在整個團隊生命週期內都是穩定的。

b221

範例 — 使用者目標對應

      "analyst": { "provider": "claude", "model": "HIGH" },
      "executor": { "provider": "claude", "model": "MEDIUM" },

標準角色

b225

orchestrator、planner、analyst、architect、executor、debugger、critic、code-reviewer、security-reviewer、test-engineer、designer、writer、code-simplifier、explore、document-specialist。

b226

使用者友善的別名會透過 normalizeDelegationRole() 正規化，例如 reviewer → code-reviewer、quality-reviewer → code-reviewer、harsh-critic → critic、build-fixer → debugger。接受的別名鍵不只在驗證時生效，在建立已解析快照與後續階段路由時也會被採用。未知角色會在解析時驗證失敗。

b227

規格欄位（TeamRoleAssignmentSpec）

b228
  • provider — "claude" | "codex" | "gemini"。省略時 → 預設為 claude。
  • model — 階級名稱（"HIGH" | "MEDIUM" | "LOW"）或明確的模型 ID。階級會透過 routing.tierModels 解析。
  • agent — 可選的 Claude agent 名稱（例如 "critic"、"executor"）。只有在解析後的提供者為 claude 時才會生效。
b229

orchestrator 被鎖定為 claude；只有 model 可由使用者設定。orchestrator 上的任何其他鍵都會被驗證器拒絕。

b230

環境變數覆寫

b231
OMC_TEAM_ROLE_OVERRIDES='{"critic":{"provider":"codex"},"code-reviewer":{"provider":"gemini"}}'
b232

優先順序：OMC_TEAM_ROLE_OVERRIDES > .claude/omc.jsonc（專案）> ~/.config/claude-omc/config.jsonc（使用者）> 內建預設值。無效的 JSON 會記錄警告並被忽略——環境變數覆寫屬於盡力而為，且絕不會中止執行。

b233

當 CLI 缺失時的回退

b234

如果在 spawn 時，已設定提供者的 CLI 不在 PATH 中，buildLaunchArgs() 會擲出例外，團隊領導者會發出可見的 SendMessage 警告，而執行階段會回退到由 buildResolvedRoutingSnapshot 預先計算出的確定性 Claude 指派（相同階級 + 相同 agent，provider: "claude"）。這種回退是刻意設計成會大聲顯示的——靜默回退會被視為測試失敗。可使用 omc doctor --team-routing 探測提供者可用性。

b235

黏著性 — 只解析一次，處處重用

b236

已解析的路由對每個團隊而言是不可變的。在團隊生命週期中途編輯設定不會影響正在執行的團隊；新的 /team 呼叫會套用新的對應。這可保證 spawn、scale-up 與 worker-restart 都會看到相同的路由，包含跨越 worktree detach 的情況（快照會隨著 TeamConfig 一起移動）。

b237

零設定行為

b238

空的 team.roleRouting 會保留修補前的行為：每個工作者都是 Claude，模型階級遵循 routing.tierModels，而 /team 3:executor ... 仍然會產生三個 Claude Sonnet 執行器。

b243

Git Worktree 整合

b244

MCP 工作者可以在彼此隔離的 git worktree 中運作，以避免並行工作者之間的檔案衝突。

b245

運作方式

b246
  1. Worktree 建立：在產生工作者之前，呼叫 createWorkerWorktree(teamName, workerName, repoRoot)，以在 .omc/worktrees/{team}/{worker} 建立一個隔離的 worktree，分支為 omc-team/{teamName}/{workerName}。
b247
  1. 工作區隔離：將 worktree 路徑作為 worker 的 BridgeConfig 中的 workingDirectory 傳入。該 worker 僅在自己的 worktree 中運作。
b248
  1. 合併協調：在 worker 完成其任務後，使用 checkMergeConflicts() 驗證分支是否能乾淨地合併，然後使用 mergeWorkerBranch() 以 --no-ff 進行合併，以保留清晰的歷史記錄。
b249
  1. 團隊清理：在團隊關閉時，呼叫 cleanupTeamWorktrees(teamName, repoRoot) 以移除所有 worktree 及其分支。
b250

API 參考

b251
FunctionDescription
createWorkerWorktree(teamName, workerName, repoRoot, baseBranch?)建立隔離的 worktree
removeWorkerWorktree(teamName, workerName, repoRoot)移除 worktree 和分支
listTeamWorktrees(teamName, repoRoot)列出所有團隊 worktree
cleanupTeamWorktrees(teamName, repoRoot)移除所有團隊 worktree
checkMergeConflicts(workerBranch, baseBranch, repoRoot)非破壞性的衝突檢查

mergeAllWorkerBranches(teamName, repoRoot, baseBranch?)合併所有已完成的 worker
b252

重要說明

b253
  • tmux-session.ts 中的 createSession() 不會處理 worktree 的建立 - worktree 的生命週期是透過 git-worktree.ts 個別管理的
  • worktree 不會在單一 worker 關閉時被清理 - 只會在團隊關閉時清理，以便事後檢視
  • 分支名稱會透過 sanitizeName() 清理，以防止注入
  • 所有路徑都會針對目錄穿越進行驗證
b254

注意事項

b255
  1. 內部任務會污染 TaskList -- 當建立一個隊友時，系統會自動建立一個具有 metadata._internal: true 的內部任務。這些會出現在 TaskList 輸出中。在計算實際任務進度時要將它們過濾掉。內部任務的主旨是隊友的名稱。
b256
  1. 沒有原子式認領 -- 與 SQLite swarm 不同，TaskUpdate 沒有交易保證。兩個隊友可能會競爭認領同一個任務。緩解方式： 領導者應在建立隊友之前，先透過 TaskUpdate(taskId, owner) 預先指派擁有者。隊友只應處理分配給他們的任務。
b257
  1. Task ID 是字串 -- ID 是自動遞增的字串（"1"、"2"、"3"），不是整數。傳入 taskId 欄位時，務必使用字串值。
b258
  1. TeamDelete 需要空團隊 -- 在呼叫 TeamDelete 之前，必須先關閉所有隊友。領導者（唯一剩餘的成員）不包含在這項檢查中。
b259
  1. 訊息會自動送達 -- 隊友訊息會以新的對話回合送達給領導者。接收訊息不需要輪詢或檢查收件匣。不過，如果領導者正處於回合中（處理中），訊息會排隊並在該回合結束時送達。
b260
  1. 隊友提示詞儲存在設定中 -- 完整的提示文字會儲存在 config.json 的 members 陣列中。不要在隊友提示詞中放入機密或敏感資料。
b261
  1. 成員會在關閉時自動移除 -- 當隊友同意關閉並終止後，會自動從 config.json 中移除。不要重新讀取設定檔並預期能找到已關閉的隊友。
b262
  1. shutdown_response 需要 request_id -- 隊友必須從傳入的關閉請求 JSON 中擷取 request_id 並回傳。格式為 shutdown-{timestamp}@{worker-name}。偽造此 ID 會導致關閉靜默失敗。
b263
  1. 團隊名稱必須是有效的 slug -- 使用小寫字母、數字和連字號。請根據任務描述推導（例如，"fix TypeScript errors" 會變成 "fix-ts-errors"）。
b264
  1. 廣播成本高 -- 每次廣播都會向每位隊友各自發送一則獨立訊息。預設請使用 message（私訊）。只有在真正全團隊範圍的關鍵警示時才使用廣播。
b265
  1. CLI worker 是一次性而非持久化的 -- Tmux CLI worker 擁有完整的檔案系統存取權限，且可以修改程式碼。不過，它們是以 автономous one-shot 工作執行 -- 不能使用 TaskList/TaskUpdate/SendMessage。領導者必須管理它們的生命週期：撰寫 prompt_file、啟動 CLI worker、讀取 output_file、標記任務完成。它們不像 Claude 隊友那樣參與團隊溝通。
