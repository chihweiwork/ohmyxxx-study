b002

<Purpose> Plan creates comprehensive, actionable work plans through intelligent interaction. It auto-detects whether to interview the user (broad requests) or plan directly (detailed requests), and supports consensus mode (iterative Planner/Architect/Critic loop with RALPLAN-DR structured deliberation) and review mode (Critic evaluation of existing plans). </Purpose>

b003

<Use_When>

b004
  • User wants to plan before implementing -- "plan this", "plan the", "let's plan"
  • User wants structured requirements gathering for a vague idea
  • User wants an existing plan reviewed -- "review this plan", --review
  • User wants multi-perspective consensus on a plan -- --consensus, "ralplan"
  • Task is broad or vague and needs scoping before any code is written

</Use_When>

b005

<Do_Not_Use_When>

b006
  • User wants autonomous end-to-end execution -- use autopilot instead
  • User wants to start coding immediately with a clear task -- use ralph or delegate to executor
  • User asks a simple question that can be answered directly -- just answer it
  • Task is a single focused fix with obvious scope -- use an execution skill instead of running it from this planning module

</Do_Not_Use_When>

b007

<Why_This_Exists> Jumping into code without understanding requirements leads to rework, scope creep, and missed edge cases. Plan provides structured requirements gathering, expert analysis, and quality-gated plans so that execution starts from a solid foundation. The consensus mode adds multi-perspective validation for high-stakes projects. </Why_This_Exists>

b008

<Execution_Policy>

b009
  • Auto-detect interview vs direct mode based on request specificity
  • Ask one question at a time during interviews -- never batch multiple questions
  • Gather codebase facts via explore agent before asking the user about them
  • Plans must meet quality standards: 80%+ claims cite file/line, 90%+ criteria are testable
  • Consensus mode runs fully automated by default; add --interactive to enable user prompts at draft review and final approval steps
  • Consensus mode uses RALPLAN-DR short mode by default; switch to deliberate mode with --deliberate or when the request explicitly signals high risk (auth/security, data migration, destructive/irreversible changes, production incident, compliance/PII, public API breakage)
  • Planning/execution boundary: planning modes inspect context and produce plans/specs/proposals only. They MUST mark artifacts as pending approval unless the user has explicitly opted into execution in the current turn or via the structured approval UI. Before explicit execution approval, planning modes MUST NOT run mutation-oriented shell commands, edit source files, commit, push, open PRs, invoke execution skills, or delegate implementation tasks.
  • Goal workflow boundary: when a plan compares Claude Code /goal, Ralph, Team, UltraQA, or artifact-only Ultragoal, identify exactly one primary loop authority and use the deterministic conflict policies refuse, adopt_existing, and artifact_only rather than non-deterministic warning handling. /goal facts must cite Claude Code/Anthropic sources only (Claude Code /goal docs: https://code.claude.com/docs/en/goal; Anthropic Claude Code changelog: https://raw.githubusercontent.com/anthropics/claude-code/main/CHANGELOG.md), and plans MUST NOT claim the /goal evaluator independently runs commands or reads files; require surfaced proof evidence before any completion claim.
  • Goal workflow doc target: for user-facing comparisons, keep examples aligned with docs/shared/mode-selection-guide.md#goal-oriented-workflow-selection and docs/REFERENCE.md#goal-workflow-ux-goal-ralph-team-ultraqa-ultragoal.

</Execution_Policy>

b013

Interview Mode (broad/vague requests)

b014
  1. Classify the request: Broad (vague verbs, no specific files, touches 3+ areas) triggers interview mode
  2. Ask one focused question using AskUserQuestion for preferences, scope, and constraints
  3. Gather codebase facts first: Before asking "what patterns does your code use?", spawn an explore agent to find out, then ask informed follow-up questions
  4. Build on answers: Each question builds on the previous answer
  5. Consult Analyst (Opus) for hidden requirements, edge cases, and risks
  6. Create plan when the user signals readiness: "create the plan", "I'm ready", "make it a work plan"
b015

Direct Mode (detailed requests)

b016
  1. Quick Analysis: Optional brief Analyst consultation
  2. Create plan: Generate comprehensive work plan immediately
  3. Review (optional): Critic review if requested
b017

Consensus Mode (--consensus / "ralplan")

b018

RALPLAN-DR modes: Short (default, bounded structure) and Deliberate (for --deliberate or explicit high-risk requests). Both modes keep the same Planner -> Architect -> Critic sequence and the same AskUserQuestion gates.

b019

Provider overrides (supported when the provider CLI is installed):

b020
  • --architect codex — replace the Claude Architect pass with omc ask codex --agent-prompt architect "..." for implementation-heavy architecture review
  • --critic codex — replace the Claude Critic pass with omc ask codex --agent-prompt critic "..." for an external review pass before execution
  • If the requested provider is unavailable, briefly note that and continue with the default Claude Architect/Critic step for that stage
b021

State lifecycle: The persistent-mode stop hook uses ralplan-state.json to enforce continuation during the consensus loop. The skill MUST manage this state:

b022
  • On entry: Call state_write(mode="ralplan", active=true, session_id=<current_session_id>) before step 1
  • On handoff to execution (approval → ralph/team): Call state_write(mode="ralplan", active=false, session_id=<current_session_id>). Do NOT use state_clear here — state_clear writes a 30-second cancel signal that disables stop-hook enforcement for ALL modes, leaving the newly launched execution mode unprotected.
  • On true terminal exit (rejection, non-interactive plan output, error/abort): Call state_clear(mode="ralplan", session_id=<current_session_id>) — no execution mode follows, so the cancel signal window is harmless.
  • Do NOT clear during intermediate steps like Critic approval or max-iteration presentation, as the user may still select "Request changes".
b023

Without cleanup, the stop hook blocks all subsequent stops with [RALPLAN - CONSENSUS PLANNING] reinforcement messages even after the consensus workflow has finished. Always pass session_id to avoid clearing other concurrent sessions' state.

b024
  1. Planner creates initial plan and a compact RALPLAN-DR summary before any Architect review. The summary MUST include:
  • Principles (3-5)
  • Decision Drivers (top 3)
  • Viable Options (>=2) with bounded pros/cons for each option
  • If only one viable option remains, an explicit invalidation rationale for the alternatives that were rejected
  • In deliberate mode: a pre-mortem (3 failure scenarios) and an expanded test plan covering unit / integration / e2e / observability
  1. User feedback _(--interactive only)_: If running with --interactive, MUST use AskUserQuestion to present the draft plan plus the RALPLAN-DR Principles / Decision Drivers / Options summary for early direction alignment with these options:
  • Proceed to review — send to Architect and Critic for evaluation
  • Request changes — return to step 1 with user feedback incorporated
  • Skip review — go directly to final approval (step 7)

If NOT running with --interactive, automatically proceed to review (step 3).

  1. Architect reviews for architectural soundness using Task(subagent_type="oh-my-claudecode:architect", ...). Architect review MUST include: strongest steelman counterargument (antithesis) against the favored option, at least one meaningful tradeoff tension, and (when possible) a synthesis path. In deliberate mode, Architect should explicitly flag principle violations. Wait for this step to complete before proceeding to step 4. Do NOT run steps 3 and 4 in parallel.
  2. Critic evaluates against quality criteria using Task(subagent_type="oh-my-claudecode:critic", ...). Critic MUST verify principle-option consistency, fair alternative exploration, risk mitigation clarity, testable acceptance criteria, and concrete verification steps. Critic MUST explicitly reject shallow alternatives, driver contradictions, vague risks, or weak verification. In deliberate mode, Critic MUST reject missing/weak pre-mortem or missing/weak expanded test plan. Run only after step 3 is complete.
  3. Re-review loop (max 5 iterations): If Critic rejects, execute this closed loop:

a. Collect all rejection feedback from Architect + Critic b. Pass feedback to Planner to produce a revised plan c. Return to Step 3 — Architect reviews the revised plan d. Return to Step 4 — Critic evaluates the revised plan e. Repeat until Critic approves OR max 5 iterations reached f. If max iterations reached without approval, present the best version to user via AskUserQuestion with note that expert consensus was not reached

  1. Apply improvements: When reviewers approve with improvement suggestions, merge all accepted improvements into the plan file before proceeding. Final consensus output MUST include an ADR section with: Decision, Drivers, Alternatives considered, Why chosen, Consequences, Follow-ups. Specifically:

a. Collect all improvement suggestions from Architect and Critic responses b. Deduplicate and categorize the suggestions c. Update the plan file in .omc/plans/ with the accepted improvements (add missing details, refine steps, strengthen acceptance criteria, ADR updates, etc.) d. Note which improvements were applied in a brief changelog section at the end of the plan

  1. On Critic approval (with improvements applied): mark the plan status as pending approval unless explicit execution approval has already been captured. _(--interactive only)_ If running with --interactive, use AskUserQuestion to present the plan with these options:
  • Approve execution via team (Recommended) — explicit opt-in to proceed via coordinated parallel team agents (/team). Team is the canonical orchestration surface since v4.1.7.
  • Approve execution via ralph — explicit opt-in to proceed via ralph+ultrawork (sequential execution with verification)
  • Compact then return for execution approval — explicit opt-in to compact the context window first (recommended when context is large after planning), then stop at the saved pending-approval plan and ask again before launching execution
  • Request changes — return to step 1 with user feedback
  • Reject — discard the plan entirely

If NOT running with --interactive, output the final plan marked pending approval, call state_clear(mode="ralplan", session_id=<current_session_id>), and stop. Do NOT auto-execute.

  1. _(--interactive only)_ User chooses via the structured AskUserQuestion UI (never ask for approval in plain text). If user selects Reject, call state_clear(mode="ralplan", session_id=<current_session_id>) and stop.
  2. On user approval (--interactive only): Call state_write(mode="ralplan", active=false, session_id=<current_session_id>) before invoking the execution skill (ralph/team), so the stop hook does not interfere with the execution mode's own enforcement. Do NOT use state_clear here — it writes a cancel signal that disables enforcement for the newly launched mode.
  • Approve execution via team: MUST invoke Skill("oh-my-claudecode:team") with the approved plan path from .omc/plans/ as context. Do NOT implement directly. The team skill coordinates parallel agents across the staged pipeline for faster execution on large tasks. This is the recommended default execution path.
  • Approve execution via ralph: MUST invoke Skill("oh-my-claudecode:ralph") with the approved plan path from .omc/plans/ as context. Do NOT implement directly. Do NOT edit source code files in the planning agent. The ralph skill handles execution via ultrawork parallel agents.
  • Compact then return for execution approval: First invoke Skill("compact") to compress the context window (reduces token usage accumulated during planning), then return with the saved pending-approval plan path and require a fresh explicit execution approval before any ralph/team launch. This path is recommended when the context window is 50%+ full after the planning session.
b025

Review Mode (--review)

b026
  1. Read plan file from .omc/plans/
  2. Evaluate via Critic using Task(subagent_type="oh-my-claudecode:critic", ...)
  3. Return verdict: APPROVED, REVISE (with specific feedback), or REJECT (replanning required)
b027

Plan Output Format

b028

Every plan includes:

b029
  • Requirements Summary
  • Acceptance Criteria (testable)
  • Implementation Steps (with file references)
  • Risks and Mitigations
  • Verification Steps
  • For consensus/ralplan: RALPLAN-DR summary (Principles, Decision Drivers, Options)
  • For consensus/ralplan final output: ADR (Decision, Drivers, Alternatives considered, Why chosen, Consequences, Follow-ups)
  • For deliberate consensus mode: Pre-mortem (3 scenarios) and Expanded Test Plan (unit/integration/e2e/observability)
b030

Plans are saved to .omc/plans/. Drafts go to .omc/drafts/. </Steps>

b031

<Tool_Usage>

b032
  • Use AskUserQuestion for preference questions (scope, priority, timeline, risk tolerance) -- provides clickable UI
  • Use plain text for questions needing specific values (port numbers, names, follow-up clarifications)
  • Use explore agent (Haiku, 30s timeout) to gather codebase facts before asking the user
  • Use Task(subagent_type="oh-my-claudecode:planner", ...) for planning validation on large-scope plans
  • Use Task(subagent_type="oh-my-claudecode:analyst", ...) for requirements analysis
  • Use Task(subagent_type="oh-my-claudecode:critic", ...) for plan review in consensus and review modes
  • CRITICAL — Consensus mode agent calls MUST be sequential, never parallel. Always await the Architect Task result before issuing the Critic Task.
  • In consensus mode, default to RALPLAN-DR short mode; enable deliberate mode on --deliberate or explicit high-risk signals (auth/security, migrations, destructive changes, production incidents, compliance/PII, public API breakage)
  • In consensus mode with --interactive: use AskUserQuestion for the user feedback step (step 2) and the final approval step (step 7) -- never ask for approval in plain text. Without --interactive, skip both prompts, mark the plan pending approval, output the final plan, and stop.
  • In consensus mode with --interactive, on explicit user approval MUST invoke Skill("oh-my-claudecode:ralph") or Skill("oh-my-claudecode:team") for execution (step 9) -- never implement directly in the planning agent
  • Before explicit execution approval, planning mode MUST NOT run mutation-oriented shell commands, edit files, commit, push, open PRs, invoke execution skills, or delegate implementation tasks; it may only inspect context and draft/update plan/spec/proposal artifacts.
  • When user selects "Compact then return for execution approval" in step 7 (--interactive only): keep the plan marked pending approval, call state_write(mode="ralplan", active=false, current_phase="pending_approval", session_id=<current_session_id>), then invoke Skill("compact") to compress the accumulated planning context. After compact, require a fresh explicit execution approval before any ralph/team launch; never auto-start implementation from compact continuation
  • CRITICAL — Consensus mode state lifecycle: Always deactivate ralplan state before stopping or handing off to execution. Use state_write(active=false) for handoff paths (approval → ralph/team) and state_clear for true terminal exits (rejection, error). Never use state_clear before launching an execution mode — its cancel signal disables stop-hook enforcement for 30 seconds.

</Tool_Usage>

b033

<Examples> <Good> Adaptive interview (gathering facts before asking):

Planner: [spawns explore agent: "find authentication implementation"]
Planner: [receives: "Auth is in src/auth/ using JWT with passport.js"]
Planner: "I see you're using JWT authentication with passport.js in src/auth/.
         For this new feature, should we extend the existing auth or add a separate auth flow?"

Why good: Answers its own codebase question first, then asks an informed preference question. </Good>

b034

<Good> Single question at a time:

Q1: "What's the main goal?"
A1: "Improve performance"
Q2: "For performance, what matters more -- latency or throughput?"
A2: "Latency"
Q3: "For latency, are we optimizing for p50 or p99?"

Why good: Each question builds on the previous answer. Focused and progressive. </Good>

b035

<Bad> Asking about things you could look up:

Planner: "Where is authentication implemented in your codebase?"
User: "Uh, somewhere in src/auth I think?"

Why bad: The planner should spawn an explore agent to find this, not ask the user. </Bad>

b036

<Bad> Batching multiple questions:

"What's the scope? And the timeline? And who's the audience?"

Why bad: Three questions at once causes shallow answers. Ask one at a time. </Bad>

b037

<Bad> Presenting all design options at once:

"Here are 4 approaches: Option A... Option B... Option C... Option D... Which do you prefer?"

Why bad: Decision fatigue. Present one option with trade-offs, get reaction, then present the next. </Bad> </Examples>

b038

<Escalation_And_Stop_Conditions>

b039
  • Stop interviewing when requirements are clear enough to plan -- do not over-interview
  • In consensus mode, stop after 5 Planner/Architect/Critic iterations and present the best version. Do NOT clear ralplan state here — the user may still select "Request changes" in the subsequent step. State is cleared only on the user's final choice (approval/rejection) or when outputting the plan in non-interactive mode.
  • Consensus mode without --interactive outputs the final plan marked pending approval and stops; with --interactive, requires explicit user approval before any implementation begins. Always call state_clear(mode="ralplan", session_id=<current_session_id>) before stopping.
  • If the user says "just do it" or "skip planning" without explicitly naming an execution path, treat it as a request to end planning: output the current plan/spec/proposal as pending approval and ask for explicit execution approval via the structured approval UI. Do NOT invoke Skill("oh-my-claudecode:ralph"), mutate files, delegate implementation, commit, push, or open a PR from the planning module until that approval exists.
  • Escalate to the user when there are irreconcilable trade-offs that require a business decision

</Escalation_And_Stop_Conditions>

b040

<Final_Checklist>

b041
  • [ ] Plan has testable acceptance criteria (90%+ concrete)
  • [ ] Plan references specific files/lines where applicable (80%+ claims)
  • [ ] All risks have mitigations identified
  • [ ] No vague terms without metrics ("fast" -> "p99 < 200ms")
  • [ ] Plan saved to .omc/plans/
  • [ ] In consensus mode: RALPLAN-DR summary includes 3-5 principles, top 3 drivers, and >=2 viable options (or explicit invalidation rationale)
  • [ ] In consensus mode final output: ADR section included (Decision / Drivers / Alternatives considered / Why chosen / Consequences / Follow-ups)
  • [ ] In deliberate consensus mode: pre-mortem (3 scenarios) + expanded test plan (unit/integration/e2e/observability) included
  • [ ] In consensus mode with --interactive: user explicitly approved before any execution; without --interactive: plan output marked pending approval only, no auto-execution
  • [ ] In consensus mode: ralplan state deactivated on every exit path — state_write(active=false) for handoff to execution, state_clear for terminal exits (rejection, error, non-interactive stop)

</Final_Checklist>

訪談模式（廣泛/模糊請求）

b014
  1. 分類請求：廣泛請求（動詞模糊、沒有特定檔案、涉及 3 個以上區域）會觸發訪談模式
  2. 提出一個聚焦問題，使用 AskUserQuestion 詢問偏好、範圍與限制
  3. 先蒐集程式碼庫事實：在詢問「你的程式碼使用哪些模式？」之前，先啟動一個 explore agent 來找出答案，然後提出有根據的後續問題
  4. 根據答案逐步推進：每個問題都建立在前一個答案之上
  5. 諮詢 Analyst（Opus）以找出隱藏需求、邊界情況與風險
  6. 建立計畫：當使用者表示已準備好時，例如「create the plan」、「I'm ready」、「make it a work plan」
b015

直接模式（詳細請求）

b016
  1. 快速分析：可選的簡短 Analyst 諮詢
  2. 建立計畫：立即產生完整的工作計畫
  3. 審查（可選）：若有要求則進行 Critic 審查
b017

共識模式（--consensus / "ralplan"）

b018

RALPLAN-DR 模式：Short（預設，具有限定結構）與 Deliberate（用於 --deliberate 或明確的高風險請求）。兩種模式都維持相同的 Planner -> Architect -> Critic 序列，以及相同的 AskUserQuestion 閘門。

b019

提供者覆寫（在已安裝提供者 CLI 時支援）：

b020
  • --architect codex — 將 Claude Architect 階段替換為 omc ask codex --agent-prompt architect "..."，用於偏重實作的架構審查
  • --critic codex — 將 Claude Critic 階段替換為 omc ask codex --agent-prompt critic "..."，用於在執行前進行外部審查
  • 如果所請求的提供者不可用，請簡要說明，並在該階段繼續使用預設的 Claude Architect/Critic 步驟
b021

狀態生命週期：持久模式的 stop hook 使用 ralplan-state.json 在共識迴圈期間強制續行。此技能必須管理這個狀態：

b022
  • 在進入時：在步驟 1 之前呼叫 state_write(mode="ralplan", active=true, session_id=<current_session_id>)
  • 在交接到執行時（核准 → ralph/team）：呼叫 state_write(mode="ralplan", active=false, session_id=<current_session_id>)。此處不要使用 state_clear —— state_clear 會寫入一個 30 秒的取消信號，會停用所有模式的 stop-hook 強制執行，讓新啟動的執行模式失去保護。
  • 在真正終止退出時（拒絕、非互動式計畫輸出、錯誤/中止）：呼叫 state_clear(mode="ralplan", session_id=<current_session_id>) —— 之後不會接續執行模式，因此這段取消信號窗口是無害的。
  • 在像 Critic 核准或達到最大迭代次數的呈現等中間步驟時，不要清除，因為使用者仍可能選擇「Request changes」。
b023

如果沒有清理，stop hook 會在共識工作流程結束後，仍以 [RALPLAN - CONSENSUS PLANNING] 強化訊息阻擋所有後續停止。請務必傳入 session_id，以避免清除其他並行 session 的狀態。

b024
  1. Planner 在任何 Architect 審查之前，先建立初始計畫與一份精簡的 RALPLAN-DR 摘要。該摘要 必須 包含：
  • Principles（3-5）
  • Decision Drivers（前三項）
  • Viable Options（>=2），並為每個選項提供有界的優缺點
  • 若只剩下一個可行選項，需為被拒絕的替代方案提供明確的 無效化理由
  • 在 deliberate mode 下：一份 pre-mortem（3 種失敗情境）與一份涵蓋 unit / integration / e2e / observability 的 expanded test plan
  1. User feedback _(--interactive only)_: 如果以 --interactive 執行，必須 使用 AskUserQuestion 呈現草案計畫，並附上 RALPLAN-DR Principles / Decision Drivers / Options 摘要以便及早對齊方向，可選擇：
  • Proceed to review — 送交 Architect 與 Critic 評估
  • Request changes — 返回步驟 1 並納入使用者回饋
  • Skip review — 直接進入最終核准（步驟 7）

如果不是以 --interactive 執行，則自動進入審查（步驟 3）。

  1. Architect 使用 Task(subagent_type="oh-my-claudecode:architect", ...) 進行架構健全性審查。Architect 審查 必須 包含：針對偏好方案最強的 steelman 反駁（antithesis）、至少一個有意義的取捨張力，以及（在可能時）一條綜合路徑。在 deliberate mode 下，Architect 應明確標示原則違反。在進行到步驟 4 之前，必須等待此步驟完成。 不要平行執行步驟 3 與 4。
  2. Critic 使用 Task(subagent_type="oh-my-claudecode:critic", ...) 依照品質標準進行評估。Critic 必須 驗證原則與選項的一致性、公平的替代方案探索、風險緩解的清晰度、可測試的驗收標準，以及具體的驗證步驟。Critic 必須 明確駁回膚淺的替代方案、與驅動因素相衝突的內容、含糊的風險，或薄弱的驗證。在 deliberate mode 下，Critic 必須 駁回缺失或薄弱的 pre-mortem，或缺失或薄弱的 expanded test plan。僅在步驟 3 完成後執行。
  3. Re-review loop（最多 5 次迭代）：如果 Critic 拒絕，執行此封閉迴圈：

a. 收集來自 Architect + Critic 的所有拒絕回饋 b. 將回饋交給 Planner 產生修訂後的計畫 c. 返回步驟 3 — Architect 審查修訂後的計畫 d. 返回步驟 4 — Critic 評估修訂後的計畫 e. 重複直到 Critic 核准，或達到最多 5 次迭代 f. 若達到最大迭代次數仍未核准，透過 AskUserQuestion 向使用者 प्रस्तुत最佳版本，並註明未達成專家共識

  1. Apply improvements：當審查者在提出改進建議的情況下核准時，將所有已接受的改進合併進計畫檔後再繼續。最終共識輸出 必須 包含一個 ADR 區段，內容包括：Decision、Drivers、Alternatives considered、Why chosen、Consequences、Follow-ups。具體而言：

a. 收集來自 Architect 與 Critic 回應中的所有改進建議 b. 去重並分類這些建議 c. 使用已接受的改進更新 .omc/plans/ 中的計畫檔（補充缺失細節、精煉步驟、強化驗收標準、更新 ADR 等） d. 在計畫末尾以簡短 changelog 區段註明已套用的改進

  1. 在 Critic 核准（且已套用改進）後：將計畫狀態標記為 pending approval，除非已明確捕獲執行核准。_（--interactive only）_ 如果以 --interactive 執行，使用 AskUserQuestion 呈現計畫，並提供以下選項：
  • Approve execution via team (Recommended) — 明確選擇透過協調式平行 team agents（/team）繼續。Team 自 v4.1.7 起是 canonical orchestration surface。
  • Approve execution via ralph — 明確選擇透過 ralph+ultrawork（含驗證的序列式執行）繼續
  • Compact then return for execution approval — 明確選擇先壓縮 context window（當規劃後 context 較大時建議），然後停在已儲存的 pending-approval 計畫並再次詢問，之後才啟動執行
  • Request changes — 返回步驟 1 並納入使用者回饋
  • Reject — 完全放棄該計畫

如果不是以 --interactive 執行，輸出標記為 pending approval 的最終計畫，呼叫 state_clear(mode="ralplan", session_id=<current_session_id>)，然後停止。不要自動執行。

  1. _（--interactive only）_ 使用者透過結構化的 AskUserQuestion UI 做出選擇（絕不要用純文字詢問核准）。如果使用者選擇 Reject，呼叫 state_clear(mode="ralplan", session_id=<current_session_id>) 並停止。
  2. 在使用者核准後（僅限 --interactive）：在啟動執行技能（ralph/team）之前，先呼叫 state_write(mode="ralplan", active=false, session_id=<current_session_id>)，以免 stop hook 影響執行模式自身的強制機制。此處不要使用 state_clear，因為它會寫入取消訊號，進而停用新啟動模式的強制機制。
  • Approve execution via team：必須 以 .omc/plans/ 中已核准的計畫路徑作為 context 呼叫 Skill("oh-my-claudecode:team")。不要直接實作。team skill 會在分階段管線中協調平行 agents，以便更快執行大型任務。這是建議的預設執行路徑。
  • Approve execution via ralph：必須 以 .omc/plans/ 中已核准的計畫路徑作為 context 呼叫 Skill("oh-my-claudecode:ralph")。不要直接實作。規劃 agent 中不要編輯 source code files。ralph skill 會透過 ultrawork parallel agents 處理執行。
  • Compact then return for execution approval：先呼叫 Skill("compact") 壓縮 context window（降低規劃期間累積的 token 用量），然後帶著已儲存的 pending-approval 計畫路徑返回，並要求在任何 ralph/team 啟動前取得新的、明確的執行核准。當規劃階段結束後 context window 已使用 50% 以上時，建議採用此路徑。

<Tool_Usage>

b032
  • 使用 AskUserQuestion 來詢問偏好問題（範圍、優先順序、時間線、風險承受度）-- 提供可點擊的 UI
  • 對於需要具體數值的問題（連接埠號、名稱、後續釐清），使用純文字
  • 在向使用者提問之前，使用 explore agent（Haiku，30s timeout）蒐集程式碼庫事實
  • 對大型範圍的計畫，使用 Task(subagent_type="oh-my-claudecode:planner", ...) 進行規劃驗證
  • 對需求分析，使用 Task(subagent_type="oh-my-claudecode:analyst", ...)
  • 在 consensus 和 review modes 中進行計畫審查，使用 Task(subagent_type="oh-my-claudecode:critic", ...)
  • CRITICAL — Consensus mode agent calls MUST be sequential, never parallel. 一定要在發出 Critic Task 之前先等候 Architect Task 的結果。
  • 在 consensus mode 中，預設使用 RALPLAN-DR short mode；在 --deliberate 或明確的高風險訊號時啟用 deliberate mode（auth/security、migrations、destructive changes、production incidents、compliance/PII、public API breakage）
  • 在帶有 --interactive 的 consensus mode 中：在使用者回饋步驟（step 2）與最終核准步驟（step 7）使用 AskUserQuestion -- 絕不要用純文字詢問核准。沒有 --interactive 時，跳過這兩個提示，將計畫標記為 pending approval，輸出最終計畫，然後停止。
  • 在帶有 --interactive 的 consensus mode 中，當使用者明確核准時，MUST 為執行（step 9）呼叫 Skill("oh-my-claudecode:ralph") 或 Skill("oh-my-claudecode:team") -- 絕不要在 planning agent 中直接實作
  • 在明確的執行核准之前，planning mode MUST NOT 執行會變更狀態的 shell commands、編輯檔案、commit、push、開啟 PR、呼叫執行技能，或委派實作任務；它只能檢視情境並草擬/更新 plan/spec/proposal artifacts。
  • 當使用者在 step 7 選擇 "Compact then return for execution approval"（僅限 --interactive）時：將計畫維持為 pending approval，呼叫 state_write(mode="ralplan", active=false, current_phase="pending_approval", session_id=<current_session_id>)，接著呼叫 Skill("compact") 以壓縮累積的 planning context。compact 之後，在任何 ralph/team 啟動之前都必須再次取得明確的執行核准；絕不要從 compact continuation 自動開始實作
  • CRITICAL — Consensus mode state lifecycle: 在停止或交接到執行之前，一律先停用 ralplan state。對於交接路徑（approval → ralph/team）使用 state_write(active=false)，對於真正的終止（rejection、error）使用 state_clear。在啟動執行模式之前絕不要使用 state_clear —— 它的取消訊號會讓 stop-hook enforcement 停用 30 秒。

</Tool_Usage>

Planner: [spawns explore agent: "find authentication implementation"]
Planner: [receives: "Auth is in src/auth/ using JWT with passport.js"]

b034

<Good> 一次只問一個問題：

Q1: "主要目標是什麼？"
A1: "提升效能"
Q2: "就效能而言，什麼更重要 -- latency 還是 throughput？"
A2: "Latency"
Q3: "針對 latency，我們是要優化 p50 還是 p99？"

為什麼好：每個問題都建立在前一個回答之上。聚焦且循序漸進。 </Good>

b035

<Bad> 詢問你可以自己查到的事情：

Planner: "你的 codebase 裡 authentication 是在哪裡實作的？"
User: "呃，我想是在 src/auth 的某個地方？"

為什麼不好：Planner 應該 spawn 一個 explore agent 來找出這個資訊，而不是問使用者。 </Bad>

b036

<Bad> 一次打包多個問題：

"範圍是什麼？時間線呢？受眾是誰？"

為什麼不好：一次問三個問題會得到很淺的回答。一次只問一個。 </Bad>

b037

<Bad> 一次呈現所有設計選項：

"這裡有 4 種做法：Option A... Option B... Option C... Option D... 你偏好哪一個？"

為什麼不好：會造成決策疲勞。先提出一個附帶取捨的選項，取得反應後，再提出下一個。 </Bad> </Examples>

<Escalation_And_Stop_Conditions>

b039
  • 當需求已清楚到足以規劃時，就停止訪談 -- 不要過度訪談
  • 在 consensus mode 中，於 5 次 Planner/Architect/Critic 迭代後停止並 प्रस्तुत 最佳版本。這裡不要清除 ralplan state —— 使用者在後續步驟中仍可能選擇 "Request changes"。只有在使用者的最終選擇（approval/rejection）時，或在 non-interactive mode 輸出計畫時，才會清除 state。
  • 沒有 --interactive 的 consensus mode 會輸出標記為 pending approval 的最終計畫並停止；有 --interactive 時，任何實作開始前都需要使用者明確核准。Always 在停止前呼叫 state_clear(mode="ralplan", session_id=<current_session_id>)。
  • 如果使用者說 "just do it" 或 "skip planning"，但沒有明確指定執行路徑，請將其視為結束規劃的請求：將目前的 plan/spec/proposal 以 pending approval 輸出，並透過結構化核准 UI 要求明確的執行核准。在該核准存在之前，絕不要從 planning module 呼叫 Skill("oh-my-claudecode:ralph")、修改檔案、委派實作、commit、push，或開啟 PR。
  • 當存在無法調和、且需要商業決策的取捨時，要升級給使用者

</Escalation_And_Stop_Conditions>
